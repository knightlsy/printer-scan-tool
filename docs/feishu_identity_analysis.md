# 飞书桌面端获取当前登录用户身份 · 技术方案分析

> 目标：在 Windows / macOS 桌面端，用户已通过飞书客户端登录的前提下，自动获取当前登录用户的用户名 / 身份标识，**且不弹出额外的登录或授权页面**。
> 分析基于本机已安装的飞书桌面端 **v7.71.12（Windows, native 客户端）** 的一手取证。

---

## 一、结论先行

**飞书桌面端并不存在一个「稳定、公开、可供任意外部桌面程序调用并回传当前登录用户」的本地 API。**

但它确实暴露了若干 **IPC / 本地通信机制**（已在安装包中实证）：

1. **URL Scheme**（`feishu://`、`lark://`、`feishu-open://` 等）—— 属于**单向「拉起 / 跳转」型 IPC**，能触发飞书执行动作 / 打开页面，但**不回传数据**给调用方。
2. **内部 JS API**（`getUserId()` / `opGetUserId()`）—— 仅能在飞书**自带 webview（小程序 / 网页应用）内**调用，外部独立进程无法访问。
3. **本地 Socket / 端口** —— 客户端内部确实开启了本地监听（见取证），但**没有公开可查的「取当前用户」端点 / 令牌机制**，且大概率需要会话令牌。
4. **本地缓存库** —— 登录身份以 `open_id` 等形式缓存在本地 SQLite，但**大概率加密存储**，且属于私有、未文档化的内部结构。

因此，能满足「已登录 + 无额外弹窗」的可行路径，实际上是**有限的几条**，下面逐一对比。

## 二、本机取证（飞书 v7.71.12, Windows）

| 证据 | 位置 | 说明 |
|---|---|---|
| `supportScheme: ["seal","rmeeting",...,"lark","feishu","feishu-open","x-feishu"]` | `7.71.12/manifest.json` | 客户端注册了多个 URL Scheme，证明 `feishu://` / `lark://` 是真实存在的 IPC 入口 |

## 三、可行方案对比

| # | 方案 | 原理 | 额外登录/授权 | 能否拿到 open_id | 稳定性 / 合规 | 适用场景 |
|---|---|---|---|---|---|---|
| **1** | **飞书开放平台 OAuth2 + 桌面端 SSO**（官方） | 注册飞书应用，用 authorization_code 换 `user_access_token`，再调 `contact` 接口取用户。桌面端已登录时，授权步骤由客户端 SSO 处理 | 首次需一次用户同意；**企业内可由管理员预先授权应用，终端用户零弹窗** | ✅ 真实 open_id / 姓名 / 邮箱 | ★★★★★ 官方、稳定、合规 | 生产环境、企业内工具（推荐） |
| **2** | **在飞书内运行（网页应用 / 小程序）** 调内部 API | 把逻辑做成飞书网页应用 / 小程序，内部用 `getUserId()` / `tt.requestAuthCode()` 取身份，再通过 `lark://` 或本地回调传给外部工具 | 需在飞书内打开该应用（一次） | ✅ | ★★★☆☆ 需上架开放平台、依赖飞书 webview | 深度飞书集成 |
| **3** | **`feishu://` 深链 + 本地回调服务**（OAuth-lite） | 你起一个本地 HTTP 服务，用 Scheme 打开飞书页面，页面识别身份后 302 到 `http://127.0.0.1:<port>/callback?user=...` | 本质仍需飞书侧页面识别身份 → 回到方案 1/2 | ✅（经飞书页面中转） | ★★☆☆☆ 脆弱、需飞书页面支撑 | 不独立成立 |
| **4** | **UI 自动化**（Windows UI Automation / macOS Accessibility） | 读取飞书窗口中已显示的姓名（侧边栏头像 / 标题） | 无；但首次可能要授权「辅助功能 / 屏幕录制」权限 | ❌ 仅展示名 | ★★☆☆☆ 布局敏感、跨版本易碎、双端实现不同 | 只取展示名 |
| **5** | **读取本地缓存库**（SQLite / 会话文件） | 解析飞书本地库中的 `open_id` / profile | 无、离线、瞬时 | ✅（若未加密） | ★☆☆☆☆ 私有未文档化 schema、疑似加密、文件锁、每次更新易碎、ToS/合规风险 | 仅个人原型 |
| **6** | **Chromium 远程调试（CDP）** | 以 `--remote-debugging-port` 启动飞书，用 DevTools 协议在渲染进程内执行 JS 调 `getUserId()` | 无，但需以调试模式重启飞书 | ✅ | ★☆☆☆☆ 需调试模式、安全/被禁风险、脆弱 | 不推荐 |
| **7** | **逆向本地 Socket / 命名管道** | 逆向 `127.0.0.1:34254` 之类端口，找 user-info 端点 + 令牌 | 无 | ✅（若找到） | ★☆☆☆☆ 私有、需令牌、版本依赖、ToS 风险 | 不推荐（除非飞书配合） |

## 四、针对你的约束的推荐

你的硬约束：**已登录 + 不弹额外登录/授权页 + Windows/macOS 桌面端**。

- **首选（合规且稳健）：方案 1 —— 飞书开放平台 OAuth2 + 桌面端 SSO。**
  关键点在于「不弹额外页面」可以通过**企业管理后台预先授权该应用（应用权限 / 通讯录权限由管理员授予）**来实现：终端用户首次使用时，由于桌面端已登录且应用已被租户授权，授权页可被客户端 SSO 静默接管，**用户体验上就是「无弹窗」**。这是唯一经官方支持、跨版本稳定、且不碰私有数据的路径。

- **完全不想接开放平台、且只需要「展示名」：方案 4 —— UI 自动化。**
  直接读飞书窗口里已经显示的用户名。实现快、零网络，但只能拿到显示名（拿到 open_id 也意义有限），且对界面布局敏感、双端要分别实现、首次要授权辅助功能权限。

- **想本地零网络瞬时拿到 open_id：方案 5 —— 读本地缓存库。**
  仅建议做个人验证 / 原型；生产环境不推荐（加密、文件锁、版本脆弱、合规风险）。

- **不推荐：** 方案 6（CDP 调试）、方案 7（逆向本地 Socket）。

---

## 五、关键澄清：「不弹额外登录授权页」的真实含义

即使走官方 SSO（方案 1），飞书 OAuth 的授权码流程在**用户首次同意应用权限**时仍会出现一次确认（可能是在桌面客户端内的一键确认，而非输入账号密码）。要做到终端用户**完全无感**，必须由**企业管理员在后台预先授予该应用所需权限**（应用免登 / 通讯录读取），这样最终用户打开工具时不需要任何额外操作。

若连「管理员预授权」都不愿做，则严格意义上**不存在既合规又完全无弹窗**的官方方案；此时只能在方案 4 / 5 之间取舍（牺牲 open_id 或牺牲稳定性/合规性）。

## 六、与当前项目的关联（备注）

当前打印机扫描工具（SCAN.GATE）已有「首次启动强制实名」的 `operator` 字段（含《百家姓》校验）。将飞书身份自动填入该字段是一个自然的增强方向：
- 走方案 1 时，可在程序启动时用 SSO 取得姓名后预填弹窗（用户仍需点一次确认，但无需手输）；
- 走方案 4 时，可静默读取展示名直接填充（但无 open_id，且依赖窗口可见）。

具体选型建议先确认：① 是否允许在飞书开放平台建一个企业内部应用；② 是否可由管理员预授权；③ 只需要展示名还是必须 open_id。
